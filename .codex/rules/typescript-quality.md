# TypeScript 与代码质量规则

## 类型系统原则
- 默认按 `strict` 模式思维编写代码。
- 禁止把 `any` 作为常规逃生口；只有在明确边界过渡时才允许短暂使用，并补注释说明原因。
- 优先使用精确类型表达领域约束，而不是“先写成宽类型再靠运行时猜”。
- 输入、输出、持久化对象、接口响应、模型结构化输出都应有清晰类型。

## 类型建模
- 区分 `数据库模型`、`领域模型`、`UI 展示模型`，不要混用一个类型走完整条链路。
- 优先使用具名类型和接口表达业务概念，避免把大量匿名对象类型直接写在函数签名上。
- 联合类型优先用于表达有限状态或分支结果，例如成功、失败、未命中。
- 对可空值做显式处理，不依赖隐式 truthy/falsy 规则混过去。

## 函数设计
- 函数名使用动词 + 领域对象，例如 `createKnowledgeBase`、`searchSourceChunks`、`buildAnswerParts`。
- 函数参数尽量少而明确；参数过多时改为具名对象。
- 返回值优先稳定结构，不随分支返回完全不同形状。
- 复杂函数拆分为“校验、转换、执行、副作用”几个具名步骤。

## 错误处理
- 预期错误与系统错误分开处理。
- 错误对象要包含足够上下文，便于日志和排查。
- 不吞错，不返回无上下文的 `null` 或裸字符串代表失败。
- 对外层调用者暴露稳定错误语义，而不是底层原始报错细节。

## 可读性约束
- 条件判断优先早返回，减少嵌套层级。
- 一段逻辑如果需要注释才能看懂，先考虑是否应该重命名或拆函数。
- 避免魔法数字和魔法字符串，提取为具名常量。
- 相同领域判断逻辑必须收敛，不在多个文件复制粘贴变体。

## 推荐实践
- Schema 先行: 用 `zod` 定义输入边界，再推导类型。
- Result 思维: 对复杂流程考虑返回显式成功/失败结构，而不是抛错和特殊值混用。
- 领域常量集中管理，例如消息状态、模型档位、支持的文档类型。

## 反模式
- 使用 `as any`、`as unknown as T` 长链断言掩盖建模问题。
- 一个类型名叫 `Data`、`Item`、`Response`，在多个模块里表达不同含义。
- 把第三方接口响应直接传给 UI，不做领域映射。
- 函数根据不同分支返回 `string | object | null | boolean` 这类不可维护的宽泛结果。
