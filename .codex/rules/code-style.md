# 编程风格规则

## 核心原则
- 高级开发风格首先体现在 `边界清晰、命名准确、状态可推导、代码易改`，而不是炫技式写法。
- 写给团队读的代码优先于写给自己看的聪明代码。
- 优先降低理解成本，再考虑局部技巧优化。

## 通用要求
- 所有源码与文档统一使用 `UTF-8` 编码。
- 默认按 `TypeScript` 严格模式思维写代码，避免滥用 `any`。
- 变量、函数、hooks、类型名都要使用语义化英文命名。
- 注释只解释原因、约束、权衡和不直观设计，不解释显而易见的动作。
- 避免超长文件和超长函数；如果一段逻辑难以命名，通常说明职责已经混杂。

## 可读性要求
- 优先早返回，减少嵌套层级。
- 一层抽象只做一类事，不在同一函数里混合校验、转换、请求、副作用和渲染决策。
- 复杂判断提取为具名变量或具名函数，不把业务规则埋在一长串条件表达式里。
- 命名要体现领域意义，例如 `citationCount` 比 `count` 更好，`retryImport` 比 `doRetry` 更好。
- 相同概念在全项目内保持同一命名，避免同义词漂移。

## 文件与模块组织
- 一个文件只表达一个主职责。
- 展示组件、业务 hooks、schema、server 服务、类型定义优先分层放置。
- 不使用无边界的 `utils.ts`、`helpers.ts` 收纳所有杂项逻辑。
- 组件文件使用 `PascalCase.tsx`，hooks 使用 `useXxx.ts`，服务与工具使用语义化英文命名。

## 状态与错误
- 状态名必须贴近业务语义，不滥用布尔值拼装复杂流程。
- 异步链路必须显式区分 `loading / success / error / empty` 等状态。
- 错误信息要便于定位问题，不能只有空泛提示。
- 流式输出、上传、索引、工具调用必须有明确中间态和恢复策略。

## UI 与样式
- 优先复用 `shadcn/ui`，在其基础上做业务定制。
- 样式优先使用 `Tailwind CSS`。
- 视觉风格遵循“企业知识工具”定位，强调清晰、可信、稳定。

## 反模式
- 为了“简洁”把多个职责压进一个函数。
- 为了“灵活”使用大量弱语义命名。
- 为了“省文件”把 schema、组件、服务、类型全部写在同一文件。
- 为了“性能”过早引入影响可读性的复杂技巧。
