# 技术选型规则

## 项目定位
- 项目是企业团队知识工作台，首版只聚焦 `文档导入、RAG 问答、引用展示、历史会话`。
- 目标是练习 AI 应用工程化与前端主导落地能力，不追求超前平台化建设。
- 技术决策遵循 `国内常用栈 + 后端简单可控 + 便于作品集展示`。

## 固定技术栈
- 前端框架: `Next.js App Router + React + TypeScript`
- UI: `shadcn/ui + Tailwind CSS`
- 前端数据层: `TanStack Query`
- 表单与校验: `react-hook-form + zod`
- AI 接入: `阿里云百炼`，Node.js 侧优先走兼容接口
- 数据库: `PostgreSQL`
- 向量检索: `pgvector`
- 对象存储: `阿里云 OSS`
- 缓存与任务: `Redis`，有明确异步场景后再补 `BullMQ`

## 技术边界
- 首版使用 `Next.js Route Handlers` 提供 API，不额外拆分独立后端。
- 不引入 `Supabase`、`Firebase` 一类一体化 BaaS 作为核心后端。
- 不因为开发速度而绕开 `PostgreSQL + pgvector` 主路线。
- 不在首版引入与目标无关的复杂基础设施，例如多租户权限平台、工作流引擎、微服务拆分。
- 新增依赖前先判断是否直接服务当前阶段目标，避免过早堆栈。

## AI 应用约束
- 消息结构统一按 `parts` 设计，而不是单一 `content: string`。
- 所有知识库回答必须支持引用来源。
- 未命中知识库时必须明确拒答，不能伪造依据。
- 工具调用按业务意图设计，不暴露任意 SQL 或任意系统执行能力给模型。
- 流式状态、工具状态、失败状态都要显式建模，不把状态混在展示文本里。

## 文档同步要求
- 技术路线调整时，同步更新 `plan.md` 与本目录相关规则。
- 如果实际实现与规则冲突，先确认是需求变化还是规则过期。
