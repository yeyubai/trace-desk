# 表单与校验规则

## 基本原则
- 表单统一使用 `react-hook-form + zod`。
- `zod schema` 是输入边界的单一事实来源，避免 UI 校验、接口校验、服务校验各写一套。
- 表单字段、接口入参、服务层输入优先围绕 schema 建模。

## 表单组织
- 简单表单可就地定义 schema 与 hook。
- 复杂业务表单应拆分 `schema`、`defaultValues`、`字段组件`、`提交逻辑`。
- 字段较多时优先按业务分组组织，而不是做成长串无层次表单。

## 校验与错误
- 错误提示要贴近字段语义，让用户知道为什么错、怎么改。
- 服务端仍需做最终校验，不能因为前端用了 zod 就省略服务端输入保护。
- 对上传、URL 导入、模型选择等关键字段做显式约束，不接受模糊自由输入。

## 提交与状态
- 提交态、禁用态、重试态必须明确。
- 长耗时操作要反馈进度或阶段，例如 `uploading`、`parsing`、`indexing`。
- 成功与失败反馈要能和具体操作对应，避免“提交失败”这种无指向提示。

## 可读性要求
- 字段命名与领域语言一致。
- 默认值集中管理，避免散落在 JSX 中。
- 表单提交函数只负责“收集已校验数据并调用用例”，不直接承担大量业务编排。

## 反模式
- 在输入框 `onChange` 中手写大量校验分支。
- 同一字段在前端、接口层、服务层使用不同名称。
- 表单提交逻辑里同时做数据转换、文件上传、轮询、路由、toast、缓存更新且不拆函数。
