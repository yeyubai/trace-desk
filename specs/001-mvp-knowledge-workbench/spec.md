## 1. 背景
当前项目目标是构建一个面向企业团队成员的 AI 知识工作台，首版聚焦：
- 文档导入
- RAG 问答
- 引用展示
- 历史会话

现有 `plan.md` 已明确技术路线和能力边界，但还缺少一份面向产品设计与交互实现的规范需求文档。  
同时，通过竞品研究可以看出，同类产品在首页结构上高度收敛：它们优先让用户“快速提问并拿到带来源的答案”，而不是先展示大量解释性模块。

因此本 spec 的目标，是把项目从“概念规划”推进到“可执行的 MVP 产品定义”。

## 2. 问题定义

### 2.1 用户问题
团队知识通常分散在文档、网页、聊天记录和业务系统中，导致：
- 用户知道信息存在，但找不到
- 用户能找到文档，但很难快速得到答案
- AI 回答如果不给来源，很难被信任
- 历史会话和沉淀反馈无法复用

### 2.2 当前机会
通过一个统一的知识问答工作台，可以把：
- 来源导入
- 检索
- 问答
- 引用追溯
- 会话沉淀

收敛到一个前端产品工作面中，降低切换成本，提高团队找答案和复用知识的效率。

## 3. 产品目标

### 3.1 MVP 目标
首版必须完成以下目标：

1. 用户可以创建并进入一个知识库工作面
2. 用户可以导入文档或网页来源
3. 用户可以基于知识库提问
4. 系统回答必须展示引用来源
5. 知识库未命中时系统必须明确拒答
6. 用户可以查看历史会话并继续追问

### 3.2 成功标准
- 用户在 1 分钟内完成第一次导入并发起提问
- 每条命中知识库的回答都能展示来源引用
- 未命中问题 100% 走明确拒答路径
- 用户能在当前会话中继续追问，无需重新组织全部上下文

## 4. 非目标
以下内容不在首版范围内：
- 多租户组织权限平台
- 复杂审批流 / 工作流引擎
- Agent Builder / 自动执行链
- 跨团队协作编辑
- 高级知识治理后台
- 接入企业 IM / CRM / 工单系统

## 5. 目标用户

### 5.1 核心用户
- 产品经理
- 客服 / 支持团队
- 运营
- 内部知识维护者

### 5.2 用户特征
- 有明确问题，但不知道答案在哪个来源里
- 需要快速得到“带出处”的答案
- 对 AI 回答的可靠性敏感
- 不愿意学习复杂的新系统

## 6. 核心价值主张
`Trace Desk` 不是普通聊天工具，而是一个“基于知识来源回答问题”的工作台。

它的首版价值是：
- 让团队把文档和网页变成可问答的知识库
- 让回答始终带来源，降低不可信感
- 让找答案、追问、回看会话在一个界面内完成

## 7. 产品原则
结合 `plan.md` 与竞品研究，MVP 采用以下原则：

1. 首页只服务一个核心任务：提问并获得带来源的答案
2. 默认展示引用，不把“引用能力”做成高级设置
3. 如果找不到依据，就直接说明，而不是编造回答
4. 搜索与聊天是连续体验，不拆成完全独立产品
5. 术语尽量面向用户任务，而不是面向系统实现
6. 次级控制放到侧栏或二级层，不抢主工作面

## 8. MVP 范围

### 8.1 页面范围
MVP 包含 4 个核心页面：

1. 知识库总览页
2. 知识导入页
3. 问答页
4. 会话与评测页

### 8.2 首页 / 主工作面要求
主页面结构按以下层级组织：

1. 顶部导航区
   - 当前知识库
   - 主搜索 / 主问答输入
   - 新建来源入口

2. 中央主工作区
   - 当前问题
   - 回答内容
   - 来源引用
   - 建议追问

3. 辅助侧栏
   - 左侧：知识库摘要 + 导入入口
   - 右侧：历史会话 / 来源详情

### 8.3 来源范围
首版支持：
- PDF
- Markdown
- TXT
- URL

不支持：
- Slack
- Confluence
- Jira
- CRM
- 工单系统

## 9. 核心用户流程

### 9.1 主路径
1. 用户进入知识库
2. 用户上传文档或输入网页链接
3. 系统解析来源并建立索引
4. 用户在主输入区提问
5. 系统先检索知识片段，再生成回答
6. 回答展示引用来源
7. 用户继续追问或切换历史会话

### 9.2 未命中路径
1. 用户提问
2. 系统未检索到足够依据
3. 回答区明确提示“当前知识库未找到可靠依据”
4. 页面仍可提供继续追问建议或建议导入相关来源

## 10. 功能需求

### 10.1 知识库总览
#### 必须有
- 展示知识库名称、描述、来源数量、分块数量、最近更新时间
- 展示当前知识库的检索状态摘要
- 可进入导入和问答

#### 应该有
- 展示关注主题 / 标签
- 展示当前知识库是否“可回答”

### 10.2 来源导入
#### 必须有
- 文件上传入口
- URL 导入入口
- 基础校验：格式、大小、URL 合法性
- 导入过程状态：
  - `idle`
  - `uploading`
  - `indexing`
  - `failed`
  - `available`

#### 应该有
- 每个来源的摘要卡片
- 最近导入来源列表
- 失败重试入口

### 10.3 问答
#### 必须有
- 主输入框
- 提交问题
- 回答列表
- 历史消息展示
- 回答引用来源
- 继续追问
- 未命中明确拒答

#### 应该有
- 模型档位切换：`Fast / Quality`
- 建议追问
- 重新生成
- 停止生成

### 10.4 来源引用
#### 必须有
- 每条回答展示引用来源
- 引用需要至少包含：
  - 来源标题
  - 引用标签
  - 摘要片段

#### 应该有
- 点击引用后查看来源详情
- 展示原文跳转入口

### 10.5 历史会话
#### 必须有
- 会话列表
- 当前会话高亮
- 显示最后更新时间
- 支持重新打开会话继续问答

#### 应该有
- 会话标题自动摘要
- 固定常用会话

### 10.6 反馈与评测
#### 必须有
- 对回答打分：正向 / 负向
- 保存反馈

#### 应该有
- 评测页查看历史反馈
- 对比不同模型档位回答效果

## 11. AI 与可信度要求

### 11.1 回答要求
- 所有知识库命中回答必须带引用
- 引用由服务端检索结果组装，不依赖模型自由生成
- 回答结构按 `parts` 模型组织

### 11.2 拒答要求
- 未命中时必须拒答
- 拒答文案必须清晰说明“没有找到可靠依据”
- 不允许伪造来源或“看起来合理”的回答

### 11.3 模型策略
- 提供 `Fast` 与 `Quality` 两档
- 档位切换不改变页面结构
- 档位属于二级控制，不应喧宾夺主

## 12. 页面与交互要求

### 12.1 信息层级
- 一级：问题输入与回答
- 二级：来源、会话、导入
- 三级：模型、运行状态、实现细节

### 12.2 文案要求
- 优先使用用户任务语言：
  - 提问
  - 来源
  - 历史会话
  - 导入
- 避免首页出现过多平台术语：
  - knowledge graph
  - orchestration
  - protocol
  - system of context

### 12.3 视觉要求
- 首页不能像营销页或产品说明页
- 需要像“日常使用中的工具”
- 保持一个明确主焦点
- 避免一屏内出现多个并列大标题

## 13. 技术映射
本 spec 对应当前技术方案：
- 前端：Next.js App Router + React + TypeScript
- UI：shadcn/ui + Tailwind CSS
- 异步状态：TanStack Query
- 表单：react-hook-form + zod
- AI：阿里云百炼兼容接口
- 数据：PostgreSQL + pgvector
- 存储：OSS
- 缓存：Redis

当前允许：
- 先用 mock 数据链路驱动 UI
- 逐步切换到 live 数据模式

## 14. 验收标准

### 14.1 功能验收
- 用户可以成功导入 1 个文件和 1 个网页链接
- 用户可以在知识库中发起问题
- 回答命中时展示至少 1 条引用
- 未命中时明确拒答
- 用户可以打开历史会话继续追问

### 14.2 体验验收
- 首页首屏在 5 秒内可理解主任务
- 用户无需学习术语就能开始提问
- 页面主视觉焦点始终落在“提问 / 回答”
- 引用查看路径清晰，不超过 1 次额外交互

### 14.3 技术验收
- API 至少包括：
  - `POST /api/knowledge/upload`
  - `POST /api/knowledge/import-url`
  - `GET /api/knowledge/:id/sources`
  - `POST /api/chat`
  - `GET /api/chat/:sessionId`
  - `POST /api/feedback`
- `mock` 与 `live` 模式边界清晰
- 回答与引用组装在服务端完成

## 15. 开放问题
- 首版首页是否采用“搜索框优先”还是“聊天输入框优先”？
- 首版会话页和评测页是否拆成独立路由，还是先内嵌在主工作台？
- 文件导入后的索引过程，首版是否只做同步 mock，再延后异步队列？
- 首版来源详情是抽屉式，还是右侧固定面板？

## 16. 版本结论
本 spec 建议首版围绕一个简洁主工作面展开，避免首页承载过多说明性内容。  
在设计和实现上，应优先接近 Glean / Notion Enterprise Search / Rovo / Guru / Onyx 这类产品的共性：一个主入口、可信回答、清晰来源、低认知负担。
