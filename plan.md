# Team Knowledge Q&A Workbench

## Next Milestone Direction (2026-03-31)

当前项目在工程上已经具备较完整的 `RAG` 基础设施，但从业务角度看，最大的短板已经转向“场景不够聚焦、失败缺少转化、治理语义不足、回答难以复用”。因此下一里程碑将优先围绕以下主线推进：

1. 将产品从“通用知识问答”收敛为“团队内部知识答疑与标准回复生成”这一默认业务场景。
2. 为来源、引用和回答补齐治理语义，让系统不只是“可回答”，还要“可管理、可更新、可追责”。
3. 把未命中与低置信结果转为知识缺口和审核任务，形成业务闭环。
4. 让 grounded answer 可以继续流向摘要、行动项和标准回复等结构化产物。
5. 在 `hit / miss / citation` 之外，补充采纳率、缺口沉淀率、复用率等业务指标。

## Default User Journey

`v1.1` 默认按以下业务路径组织页面与状态语义：
1. 用户导入资料，而不是先进入技术控制台。
2. 系统快速判断资料能否支撑团队答疑。
3. 用户直接发起首问，或请求生成标准回复草稿。
4. 回答结果以引用和可复用产物为中心展示。
5. 未命中与低置信结果转化为待补知识缺口，而不是停留在一次拒答。

## RAG Update (2026-03-29)
- 数据层已从 mock 兼容态收敛到 live-first；应用默认走真实 `PostgreSQL/pgvector`。
- 导入链路会在 chunking 后生成 embedding，并对 `thin / body-fallback / pdf-unparsed / empty` 等低质量来源做检索门控。
- 检索链路已升级为 `词法匹配 + embedding 相似度` 的混合打分，不再只依赖关键词 `includes` 命中。
- 多轮 retrieval query 已改为只利用最近用户问题做 follow-up 扩写，避免 assistant 文本噪声直接污染检索串。
- 回答链路已改为“先生成、再做证据校验、最后再流式回放”；若模型输出未通过证据校验，服务端会自动回退为拒答。
- 已补最小 RAG 评测闭环：`npm run eval:rag` 使用固定 fixture + 黄金集，覆盖 `hit / miss / citation / blocked-source`。

设计一个面向企业团队成员的“知识助手工作台”，让用户上传文档、抓取网页、建立知识库，并通过 AI 问答助手进行多轮问答、引用追溯、总结提炼和行动建议生成。  
这个题目适合用 `Next.js + shadcn/ui + 阿里百炼 SDK/兼容接口`，因为它能系统覆盖你要练的 AI 落地能力：`streaming`、Markdown 实时渲染、多模态展示、RAG、工具调用、结构化输出、性能优化、接口协作、评测与观测。

产品定位不是“聊天壳子”，而是“团队内部知识工作台”：
- 用户价值：把分散在文档和网页里的信息，变成可追溯、可问答、可复用的团队知识助手
- 练习价值：前端、AI 应用、产品思维、工程化都能练到
- 作品集价值：比普通 chat demo 更像真实业务产品

## Implementation Changes
### 产品功能与页面结构
实现 4 个核心页面，范围控制在 4-6 周可完成：
- `知识库总览页`：显示知识库列表、文档数量、最近更新时间、检索状态
- `知识导入页`：支持 PDF/Markdown/TXT 上传与网页链接抓取，展示解析进度、分块结果、失败重试
- `问答页`：支持流式回答、消息历史、引用来源、Markdown/代码块/表格渲染、停止生成、重新生成、模型切换
- `会话与评测页`：查看历史会话、收藏优质回答、对回答打分、对比不同模型或提示词效果

### 核心用户流程
按这个流程设计交互和数据流：
1. 用户创建一个知识库
2. 上传文档或输入网页链接
3. 系统完成解析、切块、向量化、入库
4. 用户在问答区中提问
5. 系统先检索知识片段，再生成流式回答
6. 回答中展示引用来源、相关文档卡片、结构化摘要
7. 用户可继续追问、切换模型、重试或反馈回答质量

### AI 能力设计
把 AI 能力做成明确的四层，而不是混在一起：
- `Streaming`：所有主回答都使用流式输出；前端处理停止生成、断流、失败重试、长对话自动滚动
- `RAG`：基于上传文档和网页内容检索，回答必须带引用来源；无命中时明确提示“未找到可靠依据”
- `Tool Calling`：至少实现 3 个工具
  - `searchKnowledgeBase`：检索知识片段
  - `fetchSourcePreview`：返回引用文档或网页摘要卡片
  - `suggestFollowups`：根据上下文生成下一步问题建议
- `Structured Output`：让模型返回统一结构，用于页面稳定渲染
  - `answer`
  - `citations`
  - `confidence`
  - `followups`

### 前端实现重点
重点把 AI 落地能力练全，而不是只把功能拼起来：
- 聊天消息结构使用 `parts` 模型，而不是单一 `content: string`
- Markdown 渲染支持代码块、表格、引用、链接、列表；对流式半截内容要有容错
- 多模态展示至少包括：文本、文件卡片、网页来源卡片、引用片段、工具执行状态
- 对长对话做性能控制：消息列表分块渲染、避免整页频繁重绘、必要时虚拟列表
- 用 `shadcn/ui` 做可复用的聊天组件、来源卡片、上传面板、状态提示、模型切换器
- 明确处理中间态：`idle`、`uploading`、`indexing`、`streaming`、`tool_running`、`failed`

### 技术与接口约束
采用以下更贴近国内常用技术栈的默认方案，保证学习收益和实现复杂度平衡：
- 前端框架：`Next.js App Router + React + TypeScript`
- UI：`shadcn/ui`
- 前端数据层：`TanStack Query`
- RAG 编排：优先采用 `LangChain.js` 统一 `Document -> split -> retrieval pipeline`，避免继续用零散 demo 逻辑拼接
- AI 接入：阿里云百炼，Node.js 侧优先使用官方兼容接口；首版可用 `openai` SDK 对接百炼兼容 API
- 数据库：`PostgreSQL`
- 向量检索：`pgvector`
- 对象存储：阿里云 `OSS`
- 鉴权：首版使用自建轻量鉴权方案；如果后续拆后端，统一收敛到服务端鉴权
- 异步任务与缓存：`Redis`，文档解析和向量化任务在需要时补 `BullMQ`
- 文档处理：首版支持 `PDF / Markdown / TXT / URL`
- 模型策略：提供 `Fast` 与 `Quality` 两档模型切换，底层映射到阿里百炼不同模型
- 关键接口：
  - `POST /api/knowledge/upload`
  - `POST /api/knowledge/import-url`
  - `GET /api/knowledge/:id/sources`
  - `POST /api/chat`
  - `GET /api/chat/:sessionId`
  - `POST /api/feedback`
- 核心实体：
  - `knowledge_base`
  - `source_document`
  - `source_chunk`
  - `chat_session`
  - `chat_message`
  - `message_citation`
  - `response_feedback`

### 企业级 RAG 设计原则
后续 RAG 继续按企业级应用口径推进，而不是 demo：

1. **导入必须可验证**
   - 用户必须知道抓到了多少正文
   - 用户必须能看到 chunk 预览
   - 用户必须知道为什么这条来源可能问不到

2. **检索必须可解释**
   - 只允许 `retrievable` 来源进入检索
   - 低质量来源要降权或明确提示
   - 未命中时不能只说“没有结果”，要能解释是否由于内容过薄、抽取回退或证据不足

3. **生成必须受证据门控**
   - 没有可用 citation 时，服务端直接拒答
   - 不允许模型伪造来源

4. **前后端必须共享同一套语义**
   - `source status`
   - `retrieval status`
   - `diagnostics`
   - `match / no-match`
   - `refusal reason`

### 最小端到端验收
每次 RAG 相关变更都至少验证这一条：

1. 导入一条网页来源
2. 查看来源详情中的 diagnostics
3. 提一个能命中该来源的问题
4. 确认回答展示引用
5. 再提一个无法命中的问题
6. 确认系统明确拒答且不伪造引用

### 4-6 周分期
按“每周都能练到新能力”的节奏推进：
1. 第 1 周：搭建基础壳子、鉴权、知识库列表、聊天页框架、基础消息状态
2. 第 2 周：接入阿里百炼流式聊天、Markdown 渲染、会话持久化、模型切换
3. 第 3 周：完成文件上传、网页抓取、切块入库、向量检索问答、引用展示
4. 第 4 周：加入工具调用、来源预览卡片、追问建议、反馈评分
5. 第 5 周：做性能优化、错误处理、空状态、基础观测面板、评测页
6. 第 6 周：补作品集质量，包括演示数据、README、录屏脚本、简历描述

## Test Plan
必须覆盖“AI 应用前端最容易出问题”的场景：
- 流式回答正常显示、停止生成、断流恢复、重新生成
- Markdown 在半截输出、代码块未闭合、表格逐步出现时仍可接受
- 上传合法文档、超大文件、损坏文件、重复导入、URL 抓取失败
- 检索命中与未命中两种路径；未命中时不能伪造引用
- 多模型切换后回答链路正常，UI 正确显示当前模型
- 工具调用成功、超时、失败三种状态都能在页面上清晰反馈
- 长会话 50+ 条消息下，页面滚动和输入仍流畅
- 会话刷新后历史消息、引用和反馈状态可恢复
- 基础评测用例至少 10 条，覆盖“事实问答”“文档总结”“无答案拒答”“引用准确性”

## Assumptions
- 项目目标是“系统锻炼 AI 落地能力 + 做成作品集项目”，不是先追求商业化
- 首版只做单租户团队工作台，不做复杂组织权限
- 首版只支持文档与网页知识源，不接企业内部 IM、CRM、工单系统
- 模型默认走 2 档选择，不做自动路由
- 文档解析与向量化允许先用同步或轻量后台任务实现，不强求复杂任务队列
- 当前技术路线优先贴近国内常见组合：`Next.js + PostgreSQL/pgvector + OSS + Redis + 阿里百炼`
- 首版以后端简单可控为目标，不引入 `Supabase` 这类一体化海外 BaaS
- UI 风格以“企业知识工具”方向为主，强调专业、清晰、可信，而不是炫技
